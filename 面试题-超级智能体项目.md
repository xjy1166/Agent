# 超级智能体（super-agent-origin）项目面试题大全

## 一、项目概述（10 题）

**Q1：请简要介绍一下这个项目是做什么的？**

>   ▎ 这是一个基于 RAG
>   ▎ 的企业知识问答系统。用户上传文档后，系统自动解析、分块、向量化存入向量数据库。用户提问时，系统通过查询改写 →
>   ▎ 知识路由 → 多通道检索（向量+关键词）→ RRF 融合 → 重排序，精准定位相关知识片段，再交给 LLM 生成带引用的答案。后端
>   ▎ Spring Boot + Vue 3，用了 PGVector/Milvus/Elasticsearch/Neo4j/MinIO/Kafka/Redis 等中间件。

**Q2：项目整体架构是什么样的？画一下架构图。**

> 后端 4 个 Maven 模块（business/common/id-generator/redisson），前端 Vue 3 SPA。中间件栈：MySQL + PostgreSQL(PGVector) + Milvus + Elasticsearch + Neo4j + MinIO + Kafka + Redis。AI 层：Spring AI + DeepSeek/Qwen3 Embedding/BGE Rerank（均通过 SiliconFlow）。

**Q3：为什么选择 RAG 而不是微调（Fine-tuning）？**

> RAG 的优势：① 数据实时更新，不需要重新训练 ② 答案可溯源，有引用文档 ③ 成本低，不需要 GPU 训练 ④ 领域知识通过检索注入，不污染模型参数。微调适合改变模型风格/行为，RAG 适合知识密集型问答。

**Q4：项目中使用了哪些核心技术和中间件？各自的作用是什么？**

| 技术 | 在本项目中做什么 |
|---|---|
| Spring Boot | 后端应用框架、REST API、自动配置 |
| Spring AI | AI 模型调用抽象、RAG 管道编排 |

**Q5：Spring AI 在项目中扮演什么角色？**

> Spring AI 是 AI 调用的抽象层，项目通过它调用 ChatModel（DeepSeek）、EmbeddingModel（Qwen3），并使用其 RAG 模块的 Query、Document、DocumentPostProcessor 等接口。Spring AI Alibaba Graph 用于智能体工作流编排。

**Q6：Spring AI 和 LangChain 有什么区别？你为什么选择 Spring AI？**

> Spring AI 是 Spring 生态原生的 AI 框架，与 Spring Boot 自动配置无缝集成，依赖注入、配置管理都用 Spring 的方式。LangChain 是 Python 生态。对于 Java 项目，Spring AI 更自然，不需要引入 Python 运行时。

**Q7：项目的服务端口是多少？前端开发服务器端口是多少？如何做代理的？**

> 后端 9082，前端 Vite dev server 5173。Vite 配置中 proxy 将 `/api` 和 `/manage` 前缀的请求代理到 `http://127.0.0.1:9082`。

**Q8：项目的许可证是什么？**

> Apache 2.0 开源许可证。

**Q9：项目的分模块设计有什么好处？**

> ① 职责分离：common 放通用工具，business 放业务逻辑，框架模块独立可复用 ② ID 生成器和 Redisson 框架可被其他项目复用 ③ 编译构建隔离，改动一个模块不影响其他 ④ 符合 Maven 多模块最佳实践。

**Q10：这个项目解决了什么业务痛点？**

> 企业内部文档分散，员工查找信息效率低。传统全文搜索只能匹配关键词，无法理解语义。本项目通过 RAG 实现语义理解 + 精确匹配的混合检索，让员工用自然语言提问就能获得准确的、有来源引用的回答。

## 二、文档处理流程（20 题）

**Q11：文档从上传到可检索，经历了哪些步骤？**

> 上传 → MinIO 存储 → Kafka 异步消息 → Tika 解析 → 文本清洗 → 结构提取 → AI 策略推荐 → 智能分块（Parent-Child）→ 向量化（Embedding）→ 写入 PGVector + ES + Neo4j。

**Q12：为什么用 Kafka 做异步处理？不用行不行？**

> 文档解析+分块+向量化是耗时操作（可能几分钟），如果同步处理，上传接口会超时。Kafka 解耦了"上传"和"处理"两个阶段，用户上传后立即返回，后台异步处理。同时 Kafka 的消息持久化保证消息不丢失，支持重试。

**Q13：Apache Tika 解析文档的原理是什么？它支持哪些格式？**

> Tika 使用自动格式检测（基于 Magic Number 和文件头），内部集成了各格式的解析器（PDFBox for PDF, POI for Office, etc.），统一封装为文本输出。支持 PDF、Word、PPT、Excel、HTML、Markdown、纯文本等几十种格式。

**Q14：Tika 解析失败了怎么办？**

> 代码中（TikaDocumentParserService 第 93-99 行）做了容错：如果是 TXT/MD 或 MIME 类型以 `text/` 开头，Tika 失败时降级为 `new String(bytes, UTF_8)` 直接读取。其他格式则抛出异常。

**Q15：文本清洗做了什么？为什么要清洗？**

> 去除 `\r\n` 统一换行符、去除空字符、压缩连续制表符/空格、限制最多 2 个连续换行。原因是 Tika 解析的原始文本包含大量格式噪声，不清洗会导致分块和检索质量下降。

**Q16：结构提取是怎么做的？为什么要提取结构？**

> 通过正则表达式识别标题信号（`#`、`1.`、`（一）` 等），构建文档的层级结构树。结构信息用于：① 按章节切块（STRUCTURE 策略）② 知识路由时定位到具体章节 ③ Neo4j 图谱查询。

**Q17：Parent-Child 分块模型是什么？为什么要用这种模型？**

> **Parent 块**（大，~2200 字符）提供完整上下文用于回答，**Child 块**（小）用于精确检索。检索时命中 Child 块，但回答时提升到 Parent 块。好处是检索精准（小块语义更集中）+ 回答完整（大块上下文更丰富）。

**Q18：递归分块（Recursive）的递归逻辑是什么？**

> 优先级：段落边界（`\n\n`）→ 行边界（`\n`）→ 句子边界（`。！？`）→ 固定窗口。每次尝试按当前粒度切分，如果单块仍然超长，则用下一级更细粒度递归切分。这保证切分尽量在语义边界处断开。

**Q19：语义分块（Semantic）是怎么判断"语义切换"的？**

> 将文本按句子分割，提取每句的 token（英文单词 + 中文字），计算相邻句子之间的 **Jaccard 相似度** = 交集/并集。当相似度低于阈值且当前块长度已够，认为主题切换，切分新块。

**Q20：LLM 分块的 Prompt 是怎么设计的？失败了怎么办？**

> Prompt 要求 LLM 将文本切成语义完整的片段，返回 JSON 数组格式。失败时回退到语义分块（第 622 行 `semanticSplit`），确保处理不会中断。

**Q21：四种分块策略（Structure/Recursive/Semantic/LLM）各适用什么场景？**

| 策略 | 适用场景 |
|---|---|
| Structure | 有清晰标题层级的文档（PDF/Word/Markdown） |
| Recursive | 长文本、无明显结构、需要控制长度 |
| Semantic | 段落丰富、主题边界明确的文档 |
| LLM | 低质量文本、结构识别不稳定、需要智能判断 |

**Q22：向量化时 batch size 为什么是 10？**

> 平衡 API 调用频率和单次请求大小。太大可能导致 SiliconFlow API 超时或 token 限制，太小则请求次数多效率低。10 是经验值，可根据 API 限速调整。

**Q23：PGVector 和 Milvus 都是向量数据库，项目为什么要同时用？**

> PGVector 是 PostgreSQL 扩展，轻量级，适合中小规模（百万级），和业务数据在同一数据库便于管理。Milvus 是专业向量数据库，适合大规模（亿级）、高并发场景。项目同时支持两者，可根据部署规模选择。

**Q24：文档画像（Document Profile）是怎么生成的？有什么用？**

> 调用 DeepSeek LLM 为文档生成摘要和关键词，存入 `super_agent_document_profile` 表。用于知识路由时快速判断某文档是否与问题相关，避免对所有文档都做向量检索。

**Q25：ES 索引初始化是怎么做的？**

> 通过 `DocumentElasticsearchIndexInitializer` 等类在应用启动时检查索引是否存在，不存在则自动创建索引映射。有 3 个初始化器分别负责文档内容索引、导航索引、知识路由索引。

**Q26：Neo4j 图谱中存储了什么？怎么查询？**

> 存储文档的结构树：节点 = 章节/步骤/列表项，边 = 包含关系/相邻关系。查询如 `findSectionByCode(documentId, "1.2.3")` 查章节编号，`findBestSection(documentId, question, "")` 语义匹配最佳章节。

**Q27：文档处理过程中的异常如何处理？**

> ① Kafka 消息失败抛异常 ② Tika 解析失败降级到直接读取 ③ LLM 分块失败降级到语义分块 ④ 每个步骤的状态记录到 `super_agent_document_task_log` ⑤ 任务有状态机（待处理/处理中/成功/失败）。

**Q28：如果文档很大（比如 100MB 的 PDF），处理会有什么问题？怎么优化？**

> 可能的问题：① Tika 解析内存溢出 ② 向量化批次太多耗时 ③ Kafka 消息可能超大。优化方案：① 流式解析 Tika ② 分批处理 ③ 大文件切片上传 ④ 异步+进度追踪。

**Q29：文档删除时怎么清理关联数据？**

> 删除 MinIO 中的原始文件和解析文本 → 删除 PGVector 中的向量数据 → 删除 ES 索引 → 删除 Neo4j 节点 → 删除 MySQL 中的文档记录、分块记录、结构节点、策略计划等。

**Q30：contentQualityLevel 是怎么评估的？有什么用？**

> 根据乱码率（`�` 字符占比）和总字符数判定：乱码率 > 2% 或 < 100 字符 → LOW，乱码率 > 0.5% 或 < 500 字符 → MEDIUM，否则 HIGH。用于策略推荐：低质量文档推荐使用 LLM 分块。

## 三、RAG 检索与问答（25 题）

**Q31：请解释 RAG 的完整工作流程。**

> Retrieval-Augmented Generation = 检索 + 增强 + 生成。① 用户提问 ② 查询改写（消除歧义）③ 知识路由（定位文档范围）④ 向量检索（语义相似度）+ 关键词检索（BM25）⑤ RRF 融合 ⑥ Rerank 精排 ⑦ 组装 Prompt（问题 + 证据 + 指令）⑧ LLM 生成答案。

**Q32：查询改写的作用是什么？什么情况下需要改写？**

> 作用：消除指代（"这个怎么用" → "Python 的 pip 怎么安装"）、补充上下文、口语转书面化。需要改写的情况：① 有对话历史且问题简短（< 18 字符）② 问题含指代词 ③ 显式多问题需要拆分。不需要的情况：问题本身已足够完整清晰。

**Q33：查询改写中"保守拆分"策略是什么意思？**

> 即使 LLM 返回 `should_split=true`，代码还会检查原文是否真的有显式多问题标记（多个 `?`、`;`、`分别`、编号列举等）。如果 LLM 想拆但代码判定"不是显式多问题"，就拒绝拆分，只用改写后的单条问题。这样防止 LLM 过度拆分导致检索碎片化。

**Q34：知识路由有几种执行模式？分别什么时候触发？**

| 模式 | 触发条件 |
|---|---|
| `GRAPH_ONLY` | 问相邻章节/目录大纲 → 直接查 Neo4j 图 |
| `GRAPH_THEN_EVIDENCE` | 问具体步骤/条目 → 先图定位再检索取证 |
| `RETRIEVAL` | 普通问答 → 混合检索 |
| `CLARIFICATION` | 问题模糊 → 澄清 |

**Q35：向量检索和关键词检索的区别是什么？各自优缺点？**

| | 向量检索 | 关键词检索 |
|---|---|---|
| 原理 | 文本 → 向量 → 余弦相似度 | BM25 算法计算词频/逆文档频率 |
| 优点 | 理解语义，"猫"能匹配"咪咪" | 精确匹配，专业术语、编号准确 |
| 缺点 | 专业术语、编号可能不准确 | 不理解同义词和语义 |

**Q36：RRF（Reciprocal Rank Fusion）融合算法的公式是什么？为什么有效？**

> `RRF_score = Σ 1/(k + rank_i)`，k=60。同一文档在不同通道排名越高，累加分数越高。有效原因：① 不需要校准不同通道的分数范围 ② 对排名敏感而非分数敏感 ③ 多通道同时排在前面的文档更可靠。

**Q37：什么是 Evidence Gate（证据闸门）？有什么作用？**

> 在 RRF 融合前，对各通道结果做分数过滤。向量通道：分数 < `minVectorSimilarity` 的直接丢弃。关键词通道：保留分数 ≥ `topScore * relativeScoreFloor` 的。作用是过滤噪声，防止低质量结果污染融合排序。

**Q38：Rerank 的原理是什么？为什么不在检索时就用 Rerank？**

> Rerank 使用交叉编码器（Cross-Encoder），同时看 query 和 document 计算相关性分数，精度高但速度慢（需逐对计算）。检索阶段用 Bi-Encoder（向量）或 BM25 做粗筛（快），Rerank 在少量候选上精排（慢但准）。两阶段设计平衡了速度和精度。

**Q39：Parent 块提升（Elevate to Parent Blocks）是怎么做的？**

> 检索命中的是 Child 块（小），通过 `documentKnowledgeService.elevateToParentBlocks()` 找到每个 Child 块关联的 Parent 块，用 Parent 块的内容替换 Child 块参与后续 Rerank 和 Prompt 组装。这样回答时有更完整的上下文。

**Q40：Prompt 组装时如果证据太多超了 token 限制怎么办？**

> `RagPromptAssemblyService` 有 token 预算管理：为每个子问题分配固定预算，按相关性排序的证据逐个填充，超出预算的证据被截断或省略。组装结果记录了 `renderedReferenceCount` 和 `omittedReferenceCount`。

**Q41：无证据时怎么处理？**

> `RagChatExecutor` 第 125-129 行：如果检索结果为空，直接返回兜底回复"当前没有足够证据支持明确回答"，不再调用 LLM。

**Q42：项目中的子问题并行检索是怎么做的？**

> `RagRetrievalEngine.retrieve()` 中，用 `CompletableFuture.supplyAsync()` + 线程池对每个子问题异步检索。每个子问题内部又对各检索通道并行调用。有超时保护（`orTimeout`），超时的子问题被忽略不影响其他。

**Q43：子问题检索超时怎么处理？**

> `orTimeout(subQuestionTimeoutMs)` + `exceptionally()` 捕获超时异常，返回空的 `SubQuestionEvidence`，在 notes 中记录"子问题 N 检索失败或超时，已自动忽略"，不影响其他子问题。

**Q44：为什么改写后的子问题数量要限制上限？**

> 每个子问题都会触发一轮完整的多通道检索+融合+重排，子问题太多会导致：① 响应时间长 ② LLM API 调用多 ③ token 消耗大。限制 `maxSubQuestions` 保证性能可控。

**Q45：检索过程中怎么记录可观测性数据？**

> 通过 `ConversationTraceRecorder`：① 每个阶段（RAG_RETRIEVE/EVIDENCE_BUDGET/ANSWER_GENERATE）记录开始/结束时间 ② 每个通道的召回数、过滤数、分数分布记录到 `ChannelExecutionView` ③ 每个 chunk 的原始分数/RRF 分数/Rerank 分数/是否进入最终 Prompt 记录到 `RetrievalResultView`。

**Q46：什么是 RAG 中的"幻觉"问题？这个项目怎么缓解？**

> 幻觉 = LLM 编造不存在的信息。缓解方式：① RAG 提供真实证据，Prompt 中明确要求"仅基于提供的证据回答" ② 无证据时返回兜底回复 ③ 答案附带引用来源，用户可验证 ④ Rerank 确保最相关的证据排在前面。

**Q47：如果用户的提问与知识库完全无关，系统怎么处理？**

> ① 检索结果为空 → 返回"当前没有足够证据" ② 检索结果分数都很低 → Evidence Gate 过滤后可能也为空 ③ OPEN_CHAT 模式下可以走通用对话不走 RAG。

**Q48：Jaccard 相似度有什么局限性？有更好的替代方案吗？**

> 局限：① 只看 token 是否出现，不考虑语义 ② 对词序不敏感 ③ 高频词会拉高相似度。更好的替代：余弦相似度（基于 TF-IDF 或 Word2Vec/Embedding）、BERTScore 等。项目中用于语义分块是够用的，因为主要是判断相邻句子主题是否一致。

**Q49：Rerank 模型和 Embedding 模型有什么区别？**

| | Embedding | Rerank |
|---|---|---|
| 类型 | Bi-Encoder（独立编码） | Cross-Encoder（联合编码） |
| 输出 | 向量（用于相似度计算） | 相关性分数 |
| 速度 | 快（一次编码，可缓存） | 慢（每对 query-doc 都要计算） |
| 用途 | 检索阶段粗筛 | Rerank 阶段精排 |

**Q50：项目的 RAG pipeline 有哪些可优化的方向？**

> ① HyDE（假设性文档嵌入）：让 LLM 先生成假设答案再检索 ② Query Expansion：扩展查询关键词 ③ 多模态检索：支持图片/表格 ④ 自适应 Top-K：根据问题复杂度动态调整 ⑤ 反馈学习：根据用户反馈优化检索参数。

**Q51：chunk 的大小对 RAG 效果有什么影响？**

> chunk 太小：上下文丢失，可能切碎完整语义单元。chunk 太大：① 语义不集中，检索相关性下降 ② 无用信息多，浪费 token 预算。Parent-Child 模型缓解了这个矛盾：Child 小块保证检索精准，Parent 大块保证回答完整。

**Q52：如果同一个 chunk 被多个子问题命中，怎么处理？**

> 每个子问题独立检索，最终结果可能有重复。`BusinessChatService.deduplicateReferences()`（第 785 行）按 `uniqueKey` 去重。Prompt 组装时也需要注意避免重复证据浪费 token。

**Q53：Embedding 模型选型有什么考量？为什么选 Qwen3？**

> 考量：① 中文效果 ② 向量维度 ③ 最大输入长度 ④ 推理速度 ⑤ API 价格。Qwen3-VL-Embedding-8B 是阿里最新的多语言 Embedding 模型，中文效果好，支持长文本，通过 SiliconFlow 调用成本可控。

**Q54：向量相似度计算用的是什么距离？余弦相似度和欧氏距离有什么区别？**

> PGVector 默认用余弦相似度（Cosine Similarity）。余弦相似度衡量方向相似性（值域 [-1,1]），对向量长度不敏感。欧氏距离衡量绝对距离，受向量长度影响。文本 Embedding 通常用余弦相似度，因为向量长度不代表信息量。

**Q55：RAG 和长上下文窗口（如 100K token）有什么关系？有了长上下文还需要 RAG 吗？**

> 长上下文不能替代 RAG：① 知识库可能有几百万 token，远超任何上下文窗口 ② 长上下文中的信息越多，LLM 的"大海捞针"问题越严重 ③ RAG 按需检索最相关的片段，成本更低 ④ 长上下文每次调用都要传输全部 token，费用高。最佳实践是 RAG + 适当长度的上下文窗口。

## 四、Spring Boot 后端（20 题）

**Q56：Spring Boot 的自动配置原理是什么？**

> `@SpringBootApplication` 包含 `@EnableAutoConfiguration`，通过 `spring.factories`（或 `AutoConfiguration.imports`）加载自动配置类，条件注解（`@ConditionalOnClass` 等）决定是否生效。

**Q57：项目的启动入口在哪里？启动时做了什么？**

> `SuperBusinessChatAgentApplication.java`。启动时：Spring Boot 自动配置 → Bean 注册 → ES 索引初始化 → Kafka Consumer 开始监听 → ReactAgent 初始化。

**Q58：MyBatis-Plus 和原生 MyBatis 有什么区别？项目中怎么用的？**

> MyBatis-Plus 在 MyBatis 基础上增强：内置 CRUD（`insert/selectById/updateById/deleteById`）、分页插件、条件构造器（`LambdaQueryWrapper`）。项目中 Mapper 继承 `BaseMapper<T>`，分页用 PageHelper。

**Q59：项目中的分页是怎么实现的？**

> 使用 PageHelper 插件，在 Mapper 查询前调用 `PageHelper.startPage(pageNo, pageSize)`，查询自动变为分页查询。封装了 `PageVo` 和 `PageUtil` 工具类。

**Q60：项目的 API 文档是怎么生成的？**

> 使用 springdoc-openapi + Knife4j。Controller 上用 `@Operation(summary="...")` 注解，`@PostMapping` 等注解自动扫描生成 OpenAPI 3.0 文档，Knife4j 提供增强的 UI 界面。

**Q61：Jackson 的序列化/反序列化在项目中做了哪些自定义？**

> common-web 模块中定义了自定义 Serializer/Deserializer，如日期格式化、Long 类型精度处理（避免 JS 精度丢失）等。

**Q62：项目中的异常处理是怎么做的？**

> `SuperAgentFrameException` 是统一业务异常，包含错误码和消息。`DocumentManageCode` 等枚举定义各业务错误码。Controller 返回统一的 `ApiResponse` 包装。

**Q63：`ApiResponse` 的设计是什么样的？**

> 统一响应格式，包含 code、message、data。`ApiResponse.ok(data)` 返回成功响应，异常时返回错误码和消息。

**Q64：Spring 的 Bean 生命周期是什么？**

> 实例化 → 属性注入 → Aware 接口回调 → BeanPostProcessor#postProcessBeforeInitialization → @PostConstruct → InitializingBean#afterPropertiesSet → init-method → BeanPostProcessor#postProcessAfterInitialization → 使用 → @PreDestroy → DisposableBean#destroy → destroy-method。

**Q65：项目中的依赖注入用了哪些方式？**

> 主要用构造器注入（推荐方式，`@AllArgsConstructor` + `final` 字段），也用 `ObjectProvider<T>` 做可选依赖（如 EmbeddingModel 可能不存在的情况）。

**Q66：`ObjectProvider` 和 `@Autowired` 有什么区别？什么场景用 ObjectProvider？**

> `@Autowired` 要求 Bean 必须存在，否则启动失败。`ObjectProvider.getIfAvailable()` 允许 Bean 不存在，返回 null。项目中用于 EmbeddingModel、ChatModel 等可能未配置的 AI 模型，实现优雅降级。

**Q67：Java 17 的新特性在项目中用到了哪些？**

> ① Record 类（`ParsedRewritePayload`、`SectionScore` 等）② switch 表达式（`switch (strategyType) { case STRUCTURE -> ... }`）③ Text Blocks（多行字符串，用于 SQL 和 Prompt 模板）④ instanceof 模式匹配。

**Q68：Text Blocks 在项目中的使用场景？**

> SQL 模板（UPSERT_SQL_TEMPLATE）和 LLM Prompt 模板（REWRITE_PROMPT）。好处是不需要手动拼接换行符和转义引号。

**Q69：Lambda 表达式和 Stream API 在项目中大量使用，有什么性能注意事项？**

> ① 避免在热循环中创建大量 Stream ② 并行 Stream（`parallelStream()`）在大数据量且无共享状态时使用 ③ `collect(Collectors.toList())` vs `toList()`（后者返回不可变列表）④ 注意 short-circuit 操作（`findFirst`/`limit`）。

**Q70：项目中的配置管理是怎么做的？**

> `application.yaml` 集中管理所有配置，通过 `@ConfigurationProperties` 绑定到 `DocumentManageProperties`、`ChatRagProperties`、`ChatAgentProperties` 等类。支持环境变量覆盖。

**Q71：Spring Boot 中 @Component、@Service、@Repository、@Controller 有什么区别？**

> 本质都是 @Component，语义不同：@Controller 处理 HTTP 请求，@Service 业务逻辑层，@Repository 数据访问层（自动转换持久化异常），@Component 通用组件。项目中按分层使用。

**Q72：项目中的线程池是怎么配置的？**

> `chatRagExecutorService` 用于 RAG 检索并行执行，`chatPostProcessExecutorService` 用于推荐问题生成等后处理。通过 `@Qualifier` 注入到具体服务中，避免线程池混用。

**Q73：SSE 和 WebSocket 有什么区别？项目为什么选 SSE？**

| | SSE | WebSocket |
|---|---|---|
| 方向 | 服务端 → 客户端（单向） | 双向 |
| 协议 | HTTP（简单） | 独立协议（复杂） |
| 断线重连 | 浏览器自动重连 | 需手动实现 |
| 适用场景 | 服务器推送（如 LLM 流式输出） | 实时双向通信（如聊天室） |

> 项目中用户提问后只需要服务端推流，不需要客户端频繁发消息，SSE 更简单。

**Q74：WebFlux 的 Flux 和 Mono 有什么区别？**

> `Mono` 代表 0-1 个元素的异步序列，`Flux` 代表 0-N 个元素。项目中 `Flux<String>` 用于 SSE 流式返回 LLM 生成的文本片段，`Mono.fromCallable()` 用于将阻塞操作（如检索）包装为异步。

**Q75：Reactor 的 Sinks.Many 是什么？在项目中怎么用的？**

> `Sinks.Many<String>` 是一个支持多元素推送的 Reactor Sink。项目中用 `Sinks.many().unicast().onBackpressureBuffer()` 创建单播 Sink，作为 SSE 的消息通道。LLM 生成的每个文本片段通过 `sink.emitNext(chunk)` 推送，完成时 `sink.emitComplete()`。

## 五、分布式与中间件（20 题）

**Q76：分布式锁的使用场景有哪些？项目中怎么用的？**

> 防止重复处理同一文档、防止索引构建并发冲突、保证定时任务单实例执行。项目中用 `@ServiceLock` 注解 + AOP 切面实现，底层用 Redisson。

**Q77：Redisson 的可重入锁原理是什么？**

> 利用 Redis 的 Hash 结构存储锁信息：key=锁名，field=线程标识，value=重入次数。加锁时 Lua 脚本原子判断，支持重入计数。Watch Dog 机制自动续期。

**Q78：分布式锁和数据库悲观锁/乐观锁有什么区别？**

| | 分布式锁 | 悲观锁 | 乐观锁 |
|---|---|---|---|
| 适用 | 跨进程/跨服务 | 单数据库 | 单数据库 |
| 实现 | Redis/ZooKeeper | SELECT FOR UPDATE | version 字段 |
| 性能 | 中等 | 低（阻塞） | 高（无锁） |
| 场景 | 分布式环境互斥 | 强一致性写操作 | 读多写少 |

**Q79：Redis 租约（Lease）和分布式锁有什么区别？**

> 锁是互斥的（同时只有一个持有者），租约是有时效的（持有者必须在 TTL 内续期，否则自动释放）。项目中用于聊天会话防并发：同一会话同时只能有一个请求在处理，TTL=30s，每 10s 续期。租约比锁更适合"防止长时间独占"的场景。

**Q80：雪花算法（Snowflake）的原理是什么？有什么优缺点？**

> 64 位 ID = 1 符号位 + 41 位时间戳 + 10 位机器 ID + 12 位序列号。优点：趋势递增、高性能、无需网络调用。缺点：依赖时钟同步（时钟回拨会生成重复 ID）、机器 ID 需要分配。

**Q81：项目中 Redis WorkerID 分配是怎么做的？**

> `RedisDisposableWorkerIdAssigner` 使用 Lua 脚本在 Redis 中原子分配。Lua 脚本保证并发安全，比数据库分配更快。支持工作数据中心 ID 分配。

**Q82：Kafka 的 Partition 有什么作用？项目中怎么用的？**

> Partition 是 Kafka 的并行单位，同一 Partition 内消息有序。项目中用 `documentId` 作为 Key，保证同一文档的消息路由到同一 Partition，消费者按序处理。多个 Partition 支持并行消费。

**Q83：Kafka 消费者怎么保证消息不丢失？**

> ① 手动提交 offset（处理成功后再提交）② 消息持久化到磁盘 ③ Producer 同步发送（`send().get()`）④ 消费失败可重试。

**Q84：MinIO 和传统文件系统/对象存储有什么区别？**

> MinIO 是 S3 兼容的对象存储，适合存储非结构化数据（文档、图片）。与文件系统相比：① 扁平命名空间（无目录层级限制）② 通过 HTTP API 访问 ③ 天然支持分布式 ④ 版本控制和生命周期管理。

**Q85：Elasticsearch 的倒排索引是什么？**

> 传统索引：文档 → 包含的词。倒排索引：词 → 出现在哪些文档。例如 `python` → [doc1, doc3, doc5]。查询时直接查倒排索引，速度极快。ES 在此基础上增加了 TF-IDF/BM25 评分、分词器、聚合等功能。

**Q86：ES 的分词器（Analyzer）是什么？中文分词怎么做？**

> Analyzer = 字符过滤器 + 分词器 + Token 过滤器。英文默认按空格分词，中文需要专门的分词器（如 IK、jieba）。项目中 ES 索引需要配置合适的中文分词器才能支持中文关键词检索。

**Q87：Neo4j 的 Cypher 查询语言是什么？举例说明项目中的查询。**

> Cypher 是图数据库查询语言，用 ASCII art 描述模式。例如查某文档的所有章节：`MATCH (s:Section)-[:BELONGS_TO]->(d:Document {id: $docId}) RETURN s`。查某章节的相邻章节：`MATCH (a:Section)-[:NEXT]->(b:Section) WHERE a.id = $sectionId RETURN b`。

**Q88：Redis 的持久化方式有哪些？RDB 和 AOF 的区别？**

> RDB：定期快照，恢复快但可能丢数据。AOF：记录每条写命令，数据安全但文件大。Redis 4.0+ 支持混合持久化（AOF + RDB）。项目中 Redis 用于缓存和锁，丢失后可重建，一般用 RDB 就够了。

**Q89：延迟队列的实现原理是什么？项目中怎么用的？**

> Redisson 的 `DelayedQueue` 底层用 Sorted Set，score = 执行时间戳。消费者轮询 ZRANGEBYSCORE 获取到期消息。项目中用于文档处理的定时重试、过期数据清理等。

**Q90：Redis Lua 脚本为什么能保证原子性？**

> Redis 执行 Lua 脚本时是单线程的，整个脚本作为一个原子操作执行，不会被其他命令打断。项目中用于 WorkerID 分配，保证并发安全。

**Q91：`@RepeatExecuteLimit` 防重复执行的实现原理是什么？**

> AOP 切面拦截方法调用，用 Redis SETNX 命令设置一个带 TTL 的 key（key = 方法签名 + 参数）。如果 SETNX 成功说明首次执行，失败说明在 TTL 内已执行过，直接拒绝或返回上次结果。

**Q92：高并发场景下 Kafka vs Redis 做消息队列怎么选？**

| | Kafka | Redis |
|---|---|---|
| 吞吐量 | 极高（百万/s） | 高（十万/s） |
| 持久化 | 磁盘持久化，可回溯 | 内存为主，可选持久化 |
| 消费模型 | Pull（消费者主动拉） | Push/Pull |
| 适用 | 大数据流、日志、事件溯源 | 缓存、简单消息、实时通知 |

**Q93：分布式系统中的一致性问题怎么解决？**

> ① 强一致性：分布式锁、2PC/TCC ② 最终一致性：消息队列、补偿机制 ③ CAP 理论：根据业务选择 CP（如 ZooKeeper）或 AP（如 Eureka）。项目中文档处理是异步的，接受最终一致性。

**Q94：Redis 集群模式有哪些？哨兵和 Cluster 的区别？**

> 哨兵：主从复制 + 哨兵监控自动故障转移，数据量受单机内存限制。Cluster：数据分片到多个节点，支持水平扩展。项目中根据数据量选择。

**Q95：Elasticsearch 和 MySQL 的区别？什么场景用 ES？**

> MySQL：关系型数据库，适合事务性操作、结构化查询。ES：搜索引擎，适合全文检索、复杂聚合、实时分析。项目中 MySQL 存业务数据（文档元数据、对话记录），ES 做全文检索和知识路由索引。

## 六、数据库设计（15 题）

**Q96：项目的数据库有哪些？各自存什么数据？**

> MySQL（19 张表）：业务数据（文档、对话、Exchange、策略、结构节点、分块等）。PostgreSQL + PGVector：向量数据。Neo4j：文档结构图谱。ES：检索索引。

**Q97：Parent-Child 分块在数据库中怎么表示的？**

> `super_agent_document_parent_block` 存父块，`super_agent_document_chunk` 存子块，子块有 `parent_block_id` 外键关联父块。向量化时只向量化子块，但检索命中后通过 `parent_block_id` 找到父块。

**Q98：对话历史在数据库中怎么存储的？**

> `super_agent_chat_dialogue` 存会话元数据，`super_agent_chat_exchange` 存每轮问答（question/answer/thinking/references），`super_agent_chat_memory_summary` 存长期压缩摘要。

**Q99：Graph Thread 和 Graph Checkpoint 表是干什么的？**

> Spring AI Alibaba Graph 的状态持久化。`GRAPH_THREAD` 存线程（会话），`GRAPH_CHECKPOINT` 存检查点（图执行状态快照）。用于 ReactAgent 的对话状态恢复。

**Q100：知识路由追踪表（knowledge_route_trace）存了什么？**

> 每次知识路由的决策过程：路由模式（GRAPH_ONLY/RETRIEVAL 等）、导航动作、定位到的章节、决策原因、查询提示等。用于分析和优化路由策略。

**Q101：检索结果快照表（chat_retrieval_result）存了什么？**

> 每次检索的详细结果：子问题、通道类型、通道排名、原始分数、RRF 分数、Rerank 分数、chunk 文本预览、是否通过闸门、是否选入最终 Prompt、淘汰原因。

**Q102：文档策略步骤表（document_strategy_step）的 strategyRole 字段代表什么？**

> `PRIMARY`（主策略，第一个步骤）、`FALLBACK`（兜底，通常是 RECURSIVE）、`OPTIMIZE`（优化，如 SEMANTIC）、`ENHANCE`（增强，如 LLM）。

**Q103：数据库索引怎么设计的？为什么要这样设计？**

> 核心索引策略：① document_id + chunk_no 联合索引（分块查询）② conversation_id + exchange_id 联合索引（对话查询）③ 任务状态索引（处理状态筛选）④ 时间字段索引（按时间排序）。

**Q104：MySQL 和 PostgreSQL 在项目中分别用在什么场景？**

> MySQL：主要业务数据存储（文档、对话、Exchange 等），使用 MyBatis-Plus。PostgreSQL + PGVector：向量数据存储，利用 PGVector 扩展的向量相似度查询能力。

**Q105：SQL 注入在项目中怎么防范的？**

> ① MyBatis-Plus 使用预编译参数（`#{}`）② 部分动态 SQL 用 `JdbcTemplate` 的 `PreparedStatement` ③ 前端输入通过 `@Valid` 校验 ④ 不拼接用户输入到 SQL。

**Q106：分表/分库在项目中有没有用到？未来如果数据量大怎么扩展？**

> 当前未分库分表。扩展方案：① PGVector 按 document_id 分区 ② MySQL 历史数据归档 ③ ES 按时间滚动索引 ④ Kafka 按 Topic 隔离不同业务线。

**Q107：数据库事务在项目中怎么管理的？**

> Spring 的 `@Transactional` 注解管理声明式事务。对于跨多个表的操作（如文档上传涉及 document 表 + task 表 + strategy 表），使用事务保证一致性。

**Q108：BaseTableData 是什么？有什么作用？**

> 基础表数据类，包含 `id`、`createTime`、`editTime` 等公共字段。所有实体类继承它，避免重复定义公共字段。MyBatis-Plus 自动填充创建时间和更新时间。

**Q109：项目的 E-R 关系是怎样的？核心实体有哪些？**

> 核心实体：Document（文档）→ DocumentStrategyPlan（策略）→ DocumentStrategyStep（步骤）→ DocumentChunk（分块）。Document → DocumentStructureNode（结构节点）。ChatDialogue（会话）→ ChatExchange（问答轮次）。KnowledgeScopeNode（知识范围）→ KnowledgeTopicNode（主题）→ Document（多对多）。

**Q110：如果要支持多租户，数据库层面需要怎么改造？**

> ① 每张表加 `tenant_id` 字段 ② MyBatis-Plus 的自动租户拦截器 ③ 查询自动加 `WHERE tenant_id = ?` ④ 向量数据按 tenant 分区 ⑤ ES 索引按租户隔离。

## 七、前端（15 题）

**Q111：Vue 3 的 Composition API 和 Options API 有什么区别？项目用的哪个？**

> Options API 按 data/methods/computed 等选项组织代码，Composition API 按逻辑功能组织代码（setup 函数）。项目用 Composition API + `<script setup>` 语法糖，代码更简洁。

**Q112：前端是怎么接收 SSE 流式数据的？**

> 用 `fetch()` + `ReadableStream`：`response.body.getReader()` 返回 `ReadableStreamDefaultReader`，循环调用 `reader.read()` 逐块读取数据，`TextDecoder` 解码后按事件类型分别处理。

**Q113：前端的 Markdown 渲染是怎么做的？安全吗？**

> `marked` 库将 Markdown 转为 HTML，`DOMPurify` 对 HTML 进行 XSS 过滤，去除危险标签和属性。`highlight.js` 做代码语法高亮。先 sanitize 再渲染，防止 XSS 攻击。

**Q114：Vue Router 的路由守卫在项目中怎么用的？**

> `adminAuth.js` 实现了管理后台的认证守卫，未登录用户访问 `/admin/*` 路由时重定向到 `/admin/login`。通过 `router.beforeEach` 全局前置守卫实现。

**Q115：Vite 和 Webpack 有什么区别？为什么选 Vite？**

> Vite 基于 ES Module 原生加载，开发环境无需打包，启动极快。Webpack 需要先打包再启动。Vite 用 Rollup 做生产打包。项目选 Vite 因为开发体验好、配置简单。

**Q116：前端的 API 层是怎么设计的？**

> `api.js` 中定义了 `chatApi` 和 `manageApi` 两个对象，封装了所有后端接口调用。SSE 接口用 `fetch` + stream，普通接口也用 `fetch`。统一处理错误和响应格式。

**Q117：前端怎么处理 LLM 返回的流式文本的逐字显示？**

> SSE 事件到达时，解析事件数据，如果是 `text` 类型就追加到当前消息的文本缓冲区。Vue 的响应式系统自动触发 UI 更新，实现"打字机"效果。

**Q118：前端的状态管理用的什么方案？**

> 项目规模不大，没有引入 Vuex/Pinia，用 Vue 3 的 `reactive`/`ref` + 组件间 props/emit 通信。会话列表、当前对话等状态在各 View 组件内管理。

**Q119：前端的代码高亮是怎么做的？**

> 使用 `highlight.js`，在 `marked` 的 renderer 中自定义代码块渲染：检测语言 → 调用 `hljs.highlight(code, {language})` → 输出高亮 HTML。

**Q120：前端怎么做请求防抖/节流的？**

> 发送消息时禁用输入框和发送按钮，防止重复提交。推荐问题点击后也禁用，等待 SSE 流结束再恢复。

**Q121：Vue 3 的响应式原理是什么？**

> 基于 `Proxy` 代理对象的 getter/setter 拦截。`reactive()` 创建响应式对象，`ref()` 创建响应式引用（内部用 `.value` 访问）。依赖收集发生在渲染时（读取属性），触发更新发生在修改属性时。

**Q122：前端的组件通信方式有哪些？项目中用了哪些？**

> ① Props/Emit（父子组件）② Provide/Inject（跨层级）③ Event Bus / mitt（兄弟组件）④ 全局状态（Pinia/Vuex）。项目主要用 Props/Emit。

**Q123：前端打包后文件太大怎么优化？**

> ① 路由懒加载（`() => import('./views/xxx.vue')`）② Tree Shaking（Vite 默认开启）③ 代码分割 ④ CDN 引入公共库 ⑤ Gzip/Brotli 压缩。

**Q124：前端怎么处理大文件上传？**

> `DocumentManageController` 用 `@RequestPart("file") MultipartFile` 接收。前端如果要做大文件上传，可用分片上传 + 断点续传（将文件切片，逐个上传，记录进度）。

**Q125：前端 Proxy 代理的原理是什么？为什么需要代理？**

> 浏览器有同源策略限制，前端 5173 端口不能直接访问后端 9082 端口。Vite dev server 的 proxy 将特定路径的请求转发到后端，对浏览器来说请求是同源的。生产环境一般用 Nginx 反向代理。

## 八、AI/LLM 相关（20 题）

**Q126：什么是 Prompt Engineering？项目中用了哪些技巧？**

> ① 角色设定（"你是企业文档问答系统的问题改写助手"）② 输出格式约束（"严格返回 JSON 数组"）③ 少样本示例（"返回格式示例：[\"片段1\"]"）④ 负面约束（"不要输出解释文字"）⑤ 分步骤指令（改写规则 1-5，拆分规则 1-6）。

**Q127：LLM 的 temperature 参数是什么？项目中怎么配置的？**

> temperature 控制输出随机性。0 = 确定性最高（适合分类/提取），1 = 随机性最高（适合创意生成）。项目中 `ChatRagProperties.RewriteOptionsProperties` 支持为改写单独配置 temperature/topP/thinking 参数。

**Q128：项目中的 LLM 调用有几种模式？**

> ① 改写：非流式调用，获取 JSON 结果 ② 答案生成：流式调用，逐 token 返回 ③ 策略推荐：非流式调用 ④ 分块：非流式调用 ⑤ 推荐问题：非流式调用 ⑥ 文档画像：非流式调用。

**Q129：为什么推荐问题生成要做异步+超时保护？**

> 推荐问题不影响核心回答，是锦上添花的功能。如果 LLM 响应慢，不应阻塞用户看到回答。异步执行 + 超时兜底确保回答完成后即使推荐生成失败也不影响用户体验。

**Q130：什么是 Embedding？向量的维度代表什么？**

> Embedding 是将文本映射到高维向量空间的过程，语义相似的文本在向量空间中距离近。维度（如 1024）代表向量的特征数量，维度越高表达能力越强但计算量越大。

**Q131：SiliconFlow 是什么？为什么用它？**

> SiliconFlow 是国内的 AI 模型推理服务平台，提供 OpenAI 兼容的 API。优势：① 国内访问速度快 ② 支持多种开源模型（DeepSeek、Qwen、BGE 等）③ 价格便宜 ④ 兼容 OpenAI API 格式，接入成本低。

**Q132：ReactAgent 是什么？在项目中起什么作用？**

> ReactAgent 是 Spring AI Alibaba 的智能体框架，实现 Reasoning + Acting 循环：观察 → 思考 → 行动 → 观察。项目中用于编排对话流程，支持工具调用（联网搜索等）。

**Q133：LLM 的 Token 是什么？中英文 token 数量差异？**

> Token 是 LLM 处理文本的最小单位。英文大约 1 token = 0.75 个单词，中文大约 1 token = 0.5-1 个字。项目中 `estimateTokenCount()` 粗估：英文单词数 + 中文字数 + 非中文字符/4。

**Q134：为什么 LLM 分块失败时要回退到语义分块而不是递归分块？**

> 语义分块考虑了语义边界（句子相似度），比纯基于字符的递归分块效果好。递归分块是最后兜底。降级链路：LLM → Semantic → Recursive。

**Q135：项目中 LLM 返回 JSON 的可靠性怎么保证？**

> ① Prompt 中明确要求返回格式 ② 给出示例 ③ 代码解析时用 `extractJsonArray()` 提取第一个 `[` 到最后一个 `]` 之间的内容（容忍多余文字）④ 解析失败时有降级方案。

**Q136：什么是 Few-shot / Zero-shot Prompting？项目中用的哪种？**

> Zero-shot：不给示例直接提问。Few-shot：给 1-3 个示例再提问。项目中主要用 Zero-shot + 格式示例（如 `["片段1","片段2"]`），属于半 few-shot。

**Q137：LLM 的 context window 是什么？项目中怎么管理 token 预算？**

> Context window 是 LLM 单次能处理的最大 token 数。项目中通过 `RagPromptAssemblyService` 管理：System Prompt + Evidence + User Question 的总 token 不能超过模型限制，超出时截断证据。

**Q138：什么是 grounding？项目中怎么实现的？**

> Grounding 是让 LLM 的回答基于事实而非臆想。项目中通过 RAG 实现：检索真实文档片段 → 嵌入 Prompt → 要求 LLM "仅基于提供的证据回答"。引用来源可验证。

**Q139：LLM 调用中的 retry 机制和熔断是怎么做的？**

> 项目中 LLM 调用失败时：① 改写失败 → 回退到规则改写 ② 分块失败 → 回退到语义分块 ③ Rerank 失败 → 回退到原始排序 ④ 推荐生成失败 → 返回空列表。每个环节都有 graceful degradation。

**Q140：Embedding 的批处理为什么要限制 batch size？**

> ① API 有单次请求 token 限制 ② 大批量可能导致超时 ③ 失败时重试成本高 ④ 内存占用。项目中 batch size = 10，可根据 API 文档调整。

**Q141：BGE Reranker 和 Cohere Rerank 有什么区别？**

> BGE（BAAI General Embedding）是开源模型，可本地部署。Cohere Rerank 是商业 API。项目用 BGE-reranker-v2-m3，通过 SiliconFlow 调用，成本低且效果好。

**Q142：大模型的 hallucination（幻觉）有哪些类型？**

> ① 事实性幻觉：编造不存在的事实 ② 忠实性幻觉：与提供的证据矛盾 ③ 输入冲突幻觉：与用户输入矛盾。RAG 主要缓解②③，通过提供真实证据约束 LLM 输出。

**Q143：项目中有没有用到 Agent 模式？和 RAG 有什么区别？**

> 项目用 Spring AI Alibaba 的 ReactAgent 框架。RAG 是"检索→生成"的单轮管道，Agent 是"思考→行动→观察"的多轮循环，可以调用工具（搜索、计算等）。Agent 适合需要多步推理的复杂问题。

**Q144：为什么要对 LLM 的输出做 post-processing？**

> ① 格式清洗（去除多余空白/标记）② JSON 提取（从混合文本中提取 JSON）③ 长度限制（截断超长输出）④ 安全过滤（去除敏感信息）⑤ 格式转换（如 Markdown 渲染）。

**Q145：项目中有没有用到向量的量化（Quantization）？**

> 目前用 float32 存储向量。未来优化可用：Scalar Quantization（float32→int8，内存省 4 倍）、Product Quantization（PQ，内存省更多但精度降低）、Binary Quantization（1-bit，极端压缩）。

**Q150：怎么评估 RAG 系统的效果？有哪些评估指标？**

**Q151：项目的 CI/CD 应该怎么设计？**

**Q152：项目的监控告警应该怎么设计？**

> ① Prometheus + Grafana：监控 JVM、API 延迟、Kafka Lag、Redis 命中率 ② ELK：日志收集和分析 ③ 自定义指标：LLM 调用次数/延迟/错误率、检索耗时、token 消耗 ④ 告警：响应时间 > 阈值、错误率 > 阈值。

**Q153：项目的缓存策略应该怎么设计？**

> ① 热门文档的检索结果缓存（Redis，TTL 1 小时）② Embedding 结果缓存（避免重复向量化）③ 文档画像缓存 ④ 会话状态缓存 ⑤ 缓存穿透/击穿/雪崩防护。

**Q154：如何设计一个 A/B 测试框架来对比不同的 RAG 策略？**

**Q155：如果要支持实时协作编辑文档（如 Google Docs），架构需要怎么改？**

> ① WebSocket 替代 SSE ② OT（Operational Transformation）或 CRDT 算法处理冲突 ③ 文档变更事件实时推送到处理管道 ④ 增量向量化（只重新向量化变更的 chunk）⑤ 版本管理。

**Q156：项目的限流策略应该怎么设计？**

> ① API 网关限流（令牌桶/漏桶）② 用户级限流（Redis 计数器）③ LLM 调用限流（并发数控制）④ 租约机制已经实现了会话级并发控制 ⑤ `@RepeatExecuteLimit` 防重复提交。

**Q157：项目的容灾和备份策略应该怎么做？**

> ① MySQL 主从 + 定时备份 ② ES 快照备份 ③ MinIO 跨区域复制 ④ Redis AOF + RDB ⑤ Kafka 数据保留策略 ⑥ Neo4j 备份 ⑦ 应用多可用区部署。

**Q158：如何设计文档处理的幂等性？**

> ① Kafka 消费者用 documentId 做幂等键 ② 向量化用 `ON CONFLICT DO UPDATE`（upsert）③ 任务状态机保证不会重复处理已完成的任务 ④ `@RepeatExecuteLimit` 防止接口重复调用。

**Q159：项目的搜索建议（Auto-complete）功能应该怎么实现？**

> ① ES Completion Suggester ② 从文档标题/章节标题构建建议索引 ③ 用户输入前缀时异步查询 ES ④ 返回 Top-N 建议 ⑤ 结合用户历史查询做个性化推荐。

**Q160：如果要支持多语言（英文文档 + 中文提问），需要改什么？**

> ① Embedding 模型换多语言模型（当前 Qwen3 已支持）② ES 分词器配置多语言 ③ Prompt 中加入语言指令（"请用中文回答"）④ 查询改写支持跨语言改写 ⑤ 文档解析支持多语言编码。

## 十、衍生/开放面试题（20 题）

**Q161：你在这个项目中遇到了什么最大的挑战？怎么解决的？**

> 示例：Parent-Child 分块的设计 — 如何在检索精度和回答完整性之间找到平衡。通过先让 LLM 推荐策略，再执行双流水线（父块大单元 + 子块小单元），检索命中子块但用父块回答。

**Q162：项目的代码质量怎么保证的？**

> ① Swagger/Knife4j API 文档自动化 ② `@Valid` 请求校验 ③ 统一异常处理 ④ Lombok 减少样板代码 ⑤ 枚举约束（策略类型、状态等）⑥ DTO/VO/Entity 分层。

**Q163：项目中用到了哪些设计模式？**

> ① 策略模式：4 种分块策略（structure/recursive/semantic/LLM）通过 switch 分发 ② 模板方法：`executePipeline()` 定义流程骨架，各策略实现具体切分 ③ 工厂模式：`ConversationExecutorRegistry` 按模式选择执行器 ④ 观察者：SSE 事件推送 ⑤ 责任链：通道过滤 → 融合 → Rerank ⑥ AOP：`@ServiceLock`、`@RepeatExecuteLimit`。

**Q164：你在这个项目中学到了什么？**

> ① RAG 系统的完整设计和调优 ② 多路检索+融合+重排的工程实践 ③ 大规模文本处理（解析→分块→向量化）的流水线设计 ④ 流式输出的前后端协作 ⑤ 分布式锁/租约在实际业务中的应用。

**Q165：如果让你重新设计这个项目，你会做哪些改变？**

> ① 引入配置中心（Nacos/Apollo）实现动态调参 ② 文档处理改用工作流引擎（如 Temporal）可视化编排 ③ 引入向量数据库中间层抽象，方便切换 Milvus/PGVector/Weaviate ④ 前端引入 Pinia 做状态管理 ⑤ 增加单元测试和集成测试覆盖率。

**Q166：怎么理解"检索增强"这三个字？**

> "检索"：根据用户问题从知识库中找到最相关的文档片段。"增强"：将找到的片段注入到 LLM 的 Prompt 中，为 LLM 提供事实依据。"生成"：LLM 基于事实依据生成回答。核心是"用检索到的知识增强 LLM 的生成能力"。

**Q167：RAG 和搜索引擎有什么区别？**

> 搜索引擎返回相关文档列表，用户自己阅读理解。RAG 在搜索的基础上增加 LLM 生成步骤，直接返回结构化答案。搜索引擎是"找到信息"，RAG 是"理解信息并回答"。

**Q168：如果你要给这个项目增加"联网搜索"功能，怎么设计？**

> 项目已有 Tavily API 依赖。设计：① 知识路由判断是否需要联网（时效性问题、知识库无结果）② 调用 Tavily Search API 获取网页内容 ③ 网页内容作为额外证据参与 Prompt 组装 ④ Prompt 中标注"以下来自网络搜索，时效性可能有限"。

**Q169：项目的文档解析能处理扫描件 PDF 吗？**

> 扫描件 PDF 是图片，Tika 无法提取文本。需要增加 OCR（光学字符识别）步骤：① Tika 检测 PDF 是否含文本层 ② 无文本层 → 调用 OCR（如 Tesseract/PaddleOCR）③ OCR 结果参与后续流程。

**Q170：怎么处理文档更新的情况？**

> 当前设计：删除旧文档 → 上传新文档 → 全量重新处理。优化方案：① 增量更新：只重新处理变更的 chunk ② 版本管理：保留历史版本 ③ Diff 检测：对比新旧文档差异，只更新差异部分。

**Q171：项目的对话记忆压缩比是多少？压缩后质量怎么保证？**

> 压缩比取决于对话长度和 LLM 摘要能力。质量保证：① 保留最近 N 轮完整对话（不压缩）② 旧对话压缩为关键信息摘要 ③ 压缩失败不覆盖原始记录 ④ 支持手动触发重建摘要。

**Q172：怎么衡量一个 RAG 系统的检索质量？**

> ① Recall@K：前 K 个结果中包含正确文档的比例 ② MRR：正确文档在结果中的排名倒数的均值 ③ Precision@K：前 K 个结果中相关文档的比例 ④ Hit Rate：至少命中一个相关文档的比例。项目中的 `retrieval_result` 表记录了完整的检索过程，可用于离线评估。

**Q173：向量数据库的 ANN（近似最近邻）搜索和精确最近邻搜索有什么区别？**

> 精确搜索遍历所有向量，100% 准确但 O(n) 复杂度，大数据量不可行。ANN 搜索用索引结构（HNSW/IVF 等）牺牲少量精度换取 O(log n) 速度。Milvus 的 HNSW 索引在 99%+ 精度下可实现毫秒级搜索。

**Q174：项目的异常信息会不会暴露给用户？**

> `BusinessChatService.buildErrorMessage()` 会提取异常的 message，但会过滤 `WebClientResponseException` 的详细信息（避免暴露内部 API 地址和响应体）。生产环境建议进一步脱敏。

**Q175：为什么项目的 API 全部用 POST 而不是 GET？**

> ① 查询参数复杂，URL 长度有限制 ② POST body 可以传 JSON ③ 某些查询涉及敏感信息不适合放 URL ④ 统一风格简化前端调用。虽然不符合 RESTful 规范，但在内部系统中是常见做法。

**Q176：项目的国际化（i18n）支持怎么样？**

> 当前硬编码中文（提示信息、Prompt 模板等）。改造方案：① 提示信息提取到 `messages.properties` ② Prompt 模板参数化语言 ③ 前端用 `vue-i18n` ④ 文档解析和 Embedding 已支持多语言。

**Q177：你对 RAG 未来的发展怎么看？**

> ① Agentic RAG：Agent 自主决定何时检索、检索什么、是否需要多轮检索 ② Multimodal RAG：支持图片、表格、视频检索 ③ Graph RAG：结合知识图谱增强推理 ④ Adaptive RAG：根据问题复杂度自动选择检索策略 ⑤ Self-RAG：LLM 自我判断是否需要检索、检索结果是否有用。

**Q178：项目中有没有做安全防护？**

> ① XSS：DOMPurify 过滤 Markdown 渲染 ② SQL 注入：预编译参数 ③ CSRF：同源策略 + Token ④ 文件上传：校验文件类型 ⑤ API 鉴权：管理后台有登录守卫 ⑥ 秘钥管理：API Key 通过配置注入，不硬编码。

**Q179：Spring AI 1.1.0 相比 1.0 有什么新特性？项目升级时注意什么？**

> 主要变化：① ChatClient API 更加 fluent ② RAG 模块增强 ③ Tool Calling 改进 ④ Observability 支持。升级注意：API 可能有 breaking changes，需要逐一检查 import 和方法签名。

**Q180：你认为这个项目最大的技术亮点是什么？**

> ① Parent-Child 双流水线分块 — 解决了检索精度和回答完整性的矛盾 ② 多通道 RRF 融合 — 兼顾语义和关键词匹配 ③ 完整的可观测性体系 — 每次检索的每一步都可追踪 ④ 知识路由 — 通过结构图谱精准定位文档范围 ⑤ 保守的查询改写 — 防止 LLM 过度拆分。

  public void handleIndexBuild(Long documentId, Long taskId, Long planId) {

      // 6.1 更新状态
      task.setCurrentStage(CHUNK_EXECUTE);
      document.setIndexStatus(BUILDING);
    
      // 6.2 从 MinIO 下载解析后的纯文本
      String parsedText = storageService.downloadText(document.getParseTextPath());
    
      // 6.3 读取确认过的策略步骤
      List<StrategyStep> stepList = listSteps(planId);
    
      // 6.4 核心：执行分块流水线
      List<ParentBlockCandidate> parentBlocks = strategyService.buildParentBlocks(
          document, plan, stepList, parsedText);
      //    流程：
      //    a. 读取结构节点（从 MySQL）
      //    b. Parent 流水线：STRUCTURE → RECURSIVE
      //       结构节点 → 父块种子 → 超长的递归再切
      //    c. Child 流水线：SEMANTIC → RECURSIVE
      //       对每个父块 → 子结构节点 → 语义切分 → 递归兜底
    
      // 6.5 数据清洗：过滤空块
      List<ParentBlockCandidate> finalParents = parentBlocks.stream()
          .filter(b -> b.getText().isNotBlank())
          .filter(b -> b.getChildChunks().stream().anyMatch(c -> c.getText().isNotBlank()))
          .toList();
    
      // 6.6 写入 MySQL
      for (ParentBlock parent : parentBlocks) {
          parentBlockMapper.insert(parent);  // 父块表
      }
      for (Chunk child : allChildren) {
          chunkMapper.insert(child);         // 子块表
      }

  回答要点：根据用户确认的策略，执行 Parent-Child 双层分块。Parent 流水线产出大块（用于回答时的完整上下文），Child
  流水线产出小块（用于检索时的精确匹配）。分块结果写入 MySQL 的父子块表。

  面试常见追问

  Q: 为什么解析和索引之间要有人工确认？

  ▎ 分块策略对检索质量影响很大。同一个文档，用结构分块可能效果很好，用递归分块可能把关键信息切碎。AI
  ▎ 推荐的策略不一定最优，用户根据实际需求确认或调整后，效果更有保障。

  Q: 如果解析失败了怎么办？

  ▎ 异常被 catch 后，文档 parseStatus 更新为 PARSE_FAILED，错误信息存入 task
  ▎ 表。用户可以在管理界面看到失败原因，重新上传。Kafka 消息已被消费，不会重复处理。

  Q: 如果向量化 API 超时了怎么办？

  ▎ 向量化失败后，catch 块会批量更新所有 chunk 的 vectorStatus 为 VECTOR_FAILED。用户可以重新触发索引构建，幂等性通过 ON
  ▎  CONFLICT DO UPDATE 保证。

  Q: 批量上传 100 个文档会怎样？

  ▎ 100 个上传请求 100ms 内全部返回。100 条 Kafka
  ▎ 消息进入队列。消费者按能力逐个处理（可配置并发数），系统不会因为瞬间请求量大而崩溃。这就是 Kafka 削峰的作用。
